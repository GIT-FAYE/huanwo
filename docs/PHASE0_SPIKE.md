**版本：** v0.1  
**日期：** 2026-06-07  
**状态：** 待执行

## 目的

在开始正式开发前，用最小验证项目（spike）确认所有关键技术风险点是否可控。**必须全部验证通过，才能进入 Phase 0 正式开发。**

---

## 验证项目

### 1. sharp + electron-builder 打包验证（⚠️ 最高优先级）

**风险**：`sharp` 是 Node.js Native Addon（依赖 `libvips` 二进制），electron-builder 打包时若未正确配置 `asar.unpack` 或 `extraResources`，会导致运行时 `Cannot find module 'sharp'` 或 `The specified module could not be found` 错误。

**验证步骤**：

1. 创建最小 Electron + electron-vite 项目
2. 安装 `sharp`（注意：需安装与 Electron Node 版本匹配的 `sharp` 预编译二进制，或用 `npm rebuild`）
3. 写一段测试代码：读取一张图片，输出其 metadata（不实际处理，只验证能加载）
4. 用 electron-builder 打包成 NSIS 安装包
5. 在**干净 Windows 虚拟机**（无 Node.js 环境）中安装并运行
6. 确认 `sharp` 能正常加载并执行

**验收标准**：打包后的应用在干净环境中能成功调用 `sharp`，无 `dlopen` / `module not found` 错误。

**若失败**：尝试以下方案（按顺序）
1. 在 `electron-builder` 配置中设置 `asar: false`（不打包 asar，简单但包体积略大）
2. 设置 `asar: { unpack: '**/*.node' }`（将 `.node` 原生二进制从 asar 中解压）
3. 或配置 `extraResources` 将 `node_modules/sharp` 原样复制到输出目录
4. 使用 `npm rebuild sharp --target=<electron-version> --runtime=electron` 重新编译
5. 换用 `jimp`（纯 JS，无 native addon，但性能较差）作为备选
6. 若以上均不可行，重新评估 Tauri 方案

---

### 2. 全局快捷键注册验证

**风险**：某些 Windows 版本或安全软件可能拦截全局快捷键注册。

**验证步骤**：

1. 用 `globalShortcut.register('Alt+Space', ...)` 注册快捷键
2. 在各应用（浏览器、VS Code、资源管理器）前台时按 `Alt+Space`，确认面板能呼出
3. 测试快捷键冲突场景（如 Windows PowerToys Run 也用 `Alt+Space`）：确认我们的注册失败时能给出提示，不影响应用启动

**验收标准**：`Alt+Space` 在主流 Windows 应用前台时可正常触发；冲突时有明确错误提示。

---

### 3. 系统托盘图标验证

**风险**：图标在浅色/深色任务栏上可能显示不清；高精度屏幕（4K）上图标模糊。

**验证步骤**：

1. 准备一个含 16×16 / 32×32 / 64×64 / 256×256 四种尺寸的单个 `.ico` 文件（多尺寸内嵌，比准备多个文件更简洁，Windows 会自动选择合适尺寸）
2. 在浅色任务栏（默认 Windows 主题）和深色任务栏（手动切换）上分别验证显示效果
3. 在高分屏（缩放 150% / 200%）上验证图标是否模糊

**验收标准**：两种主题下图标均清晰可辨；高分屏下无模糊。

---

### 4. 深色/浅色模式切换验证

**风险**：`nativeTheme` 的事件监听在某些 Windows 版本上可能不实时触发。

**验证步骤**：

1. 运行应用，手动切换 Windows 主题（设置 → 个性化 → 颜色 → 浅色/深色）
2. 确认应用面板和工具窗口实时跟随切换（无需重启应用）
3. 重启应用，确认启动时读取的是正确的系统主题

**验收标准**：主题切换实时生效；重启后主题正确。

---

### 5. 构建包在干净环境运行验证

**风险**：应用依赖了某些系统 DLL 或运行时（如 VC++ Redistributable），干净环境缺失导致无法启动。

**验证步骤**：

1. 准备一台干净 Windows 10 / 11 虚拟机（无开发环境）
2. 安装 spike 项目打包出的 NSIS 安装包
3. 运行应用，确认能正常启动并呼出面板

**验收标准**：干净环境中应用能正常启动，无缺失 DLL 错误。

---

### 6. （可选）Tauri 等效方案验证

**目的**：作为技术选型的最终确认依据。若 Tauri 方案在图片处理能力和打包体积上均优于 Electron，可切换。

**验证步骤**：

1. 创建最小 Tauri v2 项目
2. 集成图片处理（测试 `tauri-plugin-image` 或调用系统 API 处理 JPEG→WebP）
3. 验证格式覆盖率（JPEG/PNG/WebP/AVIF/GIF 动图）
4. 打包，记录安装包体积
5. 与 Electron spike 项目对比：包体积、图片处理速度、格式覆盖率

**决策依据**（以下数值为文档撰写时的**预期值**，Spike 执行后需替换为实测值）：

| 维度 | Electron（预期值，待实测） | Tauri（预期值，待实测） | 胜出方（待填写） |
|---|---|---|---|
| 安装包体积 | ~80MB | ~10MB | ❓ 实测后填写 |
| 图片格式覆盖率 | 完整（sharp） | 依赖系统 API，有盲区 | ❓ 实测后填写 |
| 开发维护成本 | 低（Web 技术栈） | 中（需 Rust 知识） | ❓ 实测后填写 |
| 全局快捷键稳定性 | 高 | 高 | 平手 |

> **注意**：上表"预期值"列数据来自规划阶段调研，**不可直接作为选型依据**。执行 Spike 后，在验证报告中填入实测数值，再做最终选型决策。

## 验证报告模板

每项验证完成后，按以下模板记录结果（**必须包含环境参数**）：

```
【验证项目】sharp + electron-builder 打包验证
【结果】✅ 通过 / ❌ 失败
【环境参数】
  - Electron 版本：28.x
  - Node.js 版本：22.x（对应 Electron 内嵌版本）
  - sharp 版本：0.33.x
  - Windows 版本：11 22H2（Build 22621）
  - 是否干净环境（无 Node.js）：是 / 否
【问题描述】（若失败）描述具体错误信息和现象
【解决方案】（若失败）记录采取的解决步骤
【结论】可以继续 / 需要换方案
```
